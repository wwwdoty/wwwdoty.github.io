HTTPS相关内容

使用HTTP的缺点

  1. 内容可能会被窃听
    解决方法:
    1) 通信加密。通过对安全套接层（SSL）和 安全传输协议(TLS)组合使用，建立安全的通信线路。
    2) 内容加密。对HTTP报文进行加密。
  2. 不能确认通信双方的身份
    伪装客户端，伪装服务器，特定用户权限信息，无法阻止海量请求下的DoS攻击（拒绝服务攻击）
    解决方法：SSL使用证书确定对方身份
  3. 无法证明报文完整性，判断信息是否准确
    在请求或响应传输途中遭到拦截并篡改内容的中间人攻击（MITM）
    解决方法:MD5和SHA-1等散列值校验的算法，数字签名方法，需要客户端用户自行检查。

HTTPS = HTTP + 通信加密 + 认证 + 完整性保护

采用SSL，使HTTP拥有加密、证书和完整性保护的功能，把这样的HTTP称为HTTPS，HTTPS先和SSL通信，再由SSL和TCP通信。

SSL位于应用层和传输层之间，它能够为不论什么基于TCP等可靠连接的应用层协议提供安全性保证。

  1. 加密方法

    • 共享密钥加密/对称密钥加密：加密解密同用一个密钥。

    • 公开密钥加密：发送方使用对方的公开密钥进行加密，对方收到信息后用自己的私有密钥解密。SSL通常採用RSA算法加密传输密钥。

      HTTPS采用混合加密机制，用公开密钥加密方式加密发送密钥，确保密钥安全下，使用共享密钥加密方式进行通信。

  2. 证明公开密钥正确性的证书
    使用由数字证书认证机构（CA）和其相关机构颁发的公开密钥证书。流程如下：
    1) 服务器:向数字证书认证机构提出公开密钥的申请。
    2) CA: 用自己的私有密钥对已申请的公开密钥做数字签名，将该公开密钥放入公钥证书（数字证书）并绑定
    3) 服务器: 将数字证书认证机构颁发的公钥证书发给客户端
    4) 客户端:使用收到的公开密钥，确认公开密钥的真实性。

    • 多数浏览器开发商会事先在内部植入常用认证机关的公开密钥。
    • EV SSL证书: 确认服务器背后运营企业是否真实存在
    • 使用客户端证书，可确认客户端真实，但存在自行安装付费，不能确认用户本人等问题。

  3. 消息完整性
    SSL利用基于MD5或SHA的MAC算法来保证消息完整性。
    MAC算法:在密钥参与下的数据摘要算法，将密钥和随意长度的数据转换为固定长度数据。MAC算法计算出MAC值附在消息后发送给对方，对方接收后再计算MAC值，若相同说明消息完整。

HTTPS安全通信机制

SSL通过握手过程在客户端和服务器之间交换参数，建立连接。

  1. 客户端：发送Client hello消息将它支持的SSL版本号、加密算法、密钥交换算法、MAC算法等信息发送给客户端。
  2. 服务器：发送Server Hello消息，确定本次通信採用的SSL版本号和加密套件。
  3. 服务器：发送Certificate消息给客户端，报文中是包含公钥信息的公钥证书
  4. 服务器：发送Server Hello done给客户端，通知SSL版本号和加密套件协商结束。
  5. 客户端：验证证书合法后，利用证书中的公钥加密客户端随机生成的premaster secret，并通过Client Key Exchange消息发送给服务器。
  6. 客户端：发送Change Cipher Spec消息，通知服务器此后的通信将采用协商好的密钥和加密套件进行加密。
  7. 客户端计算除Change Cipher Spec消息外全部已交互的消息）的Hash值，利用协商好的密钥和加密套件处理Hash值（计算并加入MAC值、加密等），并通过Finished消息发送给服务器。利用相同的方法计算已交互的握手消息的Hash值，并与Finished消息的解密结果比較，假设二者相同，且MAC值验证成功，则证明密钥和加密套件协商成功。
  8. 服务器：同样发送 Change Cipher Spec 报文
  9. 服务器：服务器同样发送Finished报文。客户端利用相同的方法计算已交互的握手消息的Hash值，并与Finished消息的解密结果比較，假设二者相同。且MAC值验证成功。则证明密钥和加密套件协商成功。
  10. 服务器和客户端的 Finished 报文交换完毕之后，SSL 连接
    就算建立完成。当然，通信会受到 SSL 的保护。从此处开始进行应用
    层协议的通信，即发送 HTTP 请求。
